## 1. 引言

### 1.1 编写目的

本文档旨在详细描述**高校科研项目管理系统**的功能需求、非功能需求及系统约束，为系统的开发、测试、部署和维护提供明确的标准和依据。

### 1.2 项目背景

高校科研管理涉及项目申报、专家评审、经费管理、成果登记、验收结题等多个环节，传统的手工或半手工管理方式存在信息孤岛、流程不规范、数据统计困难等问题。本系统旨在为高校科研处、财务处、科研人员和评审专家提供一个统一的、覆盖科研项目全生命周期的信息化管理平台。

### 1.3 适用范围

- **目标用户**：高校科研处管理人员、财务处管理人员、科研人员（教师/研究员）、评审专家
- **适用场景**：高校及科研院所的科研项目全流程管理

### 1.4 术语定义

| 术语 | 说明 |
|------|------|
| 纵向项目 | 由政府财政资金资助的科研项目（国家级、省部级、市厅级） |
| 横向项目 | 由企事业单位委托或合作的科研项目 |
| 形式审查 | 对项目申报材料的完整性、规范性进行的初步审核 |
| 立项公示 | 专家评审完成后，对拟立项项目进行公示的阶段 |
| 预算执行率 | 已支出金额 / 预算总额 × 100% |
| 结余经费 | 项目结束后未使用完的经费余额 |

## 2. 总体描述

### 2.1 系统概述

本系统是一个基于 B/S 架构的 Web 应用，采用前后端一体化设计。后端使用 Python Flask 框架提供 RESTful API，前端使用 Bootstrap + jQuery 构建单页应用（SPA），数据存储采用 MySQL 关系型数据库。

### 2.2 用户角色

| 角色 | 标识 | 职责描述 |
|------|------|----------|
| **科研人员** | `科研人员` | 项目申报、预算编制、进展报告提交、验收申请、成果登记、经费报销申请 |
| **评审专家** | `专家` | 接收分配的评审任务，对项目进行打分和出具评审意见 |
| **科研处管理员** | `科研处` | 发布申报通知、形式审查、分配评审专家、立项审批、验收评审、证书发放、数据统计 |
| **财务处管理员** | `财务处` | 经费到账登记、支出报销审批、经费执行率监控、结余经费处理 |

### 2.3 运行环境

| 项目 | 要求 |
|------|------|
| 服务器操作系统 | Windows / Linux / macOS |
| Python 版本 | 3.8 及以上 |
| MySQL 版本 | 5.7 及以上 或 8.0 |
| 浏览器 | Chrome、Firefox、Edge 等现代浏览器 |

## 3. 功能需求

### 3.1 首页仪表盘 (Dashboard)

**FR-1.1 通知公告展示**
- 系统应按发布时间倒序展示所有通知公告
- 显示通知标题、内容摘要和发布日期

**FR-1.2 即将到期项目提醒**
- 系统应展示 30 天内即将到期且未完成验收的项目
- 显示项目编号、名称、剩余天数
- 按剩余天数升序排列

**FR-1.3 快速入口**
- 提供项目申报、经费到账、成果登记的快捷导航按钮

---

### 3.2 项目申报管理

**FR-2.1 发布申报通知**
- 科研处管理员可发布项目申报通知
- 通知包含标题、内容、发布日期
- 自动记录发布人

**FR-2.2 在线项目申报**
- 科研人员可在线填写项目申报书
- 必填字段：项目编号、项目名称、项目类型（纵向/横向）、级别、负责人工号、开始日期、结束日期、总预算
- 支持添加多个预算科目（科目名称 + 金额）
- 支持上传附件文件
- 提交后项目状态自动设为"申报中"

**FR-2.3 形式审查**
- 科研处管理员对申报项目进行形式审查
- 审查结果：通过 / 不通过
- 通过 → 状态变更为"专家评审"
- 不通过 → 状态退回"申报中"

**FR-2.4 专家分配与评审**
- 科研处管理员为一个项目分配多位评审专家
- 专家提交评审意见（结果、打分 0-100、评语）
- 系统自动检测所有专家是否完成评审
- 所有专家评审完成后，项目状态自动变更为"立项公示"

**FR-2.5 立项审批**
- 科研处管理员对立项公示项目进行最终审批
- 审批通过 → 状态变更为"已立项"

---

### 3.3 项目执行管理

**FR-3.1 合同/任务书管理**
- 支持上传合同文件（文件路径、签署日期、合同内容）
- 合同与项目一对一关联

**FR-3.2 预算编制与执行监控**
- 展示项目各预算科目的预算金额、已支出金额、执行率
- 调用数据库函数进行三级预警：正常（<85%）、预警（≥85%）、超支（≥100%）
- 预警科目以红色标识

**FR-3.3 年度进展报告**
- 科研人员按年度提交项目进展报告
- 报告信息：项目编号、报告年度、提交日期、报告内容
- 提交后初始状态为"待审核"

**FR-3.4 重要事项变更申请**
- 支持三种变更类型：预算调整、延期、成员变更
- 变更申请记录原值、新值、变更理由
- 提交后初始状态为"待审批"

---

### 3.4 项目验收管理

**FR-4.1 验收申请**
- 科研人员提交验收申请
- 需上传结题材料附件
- 提交后项目状态变更为"验收申请"

**FR-4.2 验收评审**
- 科研处管理员进行验收评审
- 评审结果：通过 / 不通过
- 通过 → 项目状态变更为"验收通过"，触发器自动记录验收日期
- 不通过 → 项目状态退回"执行中"

**FR-4.3 结题证书发放**
- 科研处管理员为验收通过的项目发放结题证书
- 支持上传证书文件链接

---

### 3.5 经费管理

**FR-5.1 经费到账登记**
- 财务处管理员登记项目经费到账
- 信息：项目编号、到账金额、到账日期、用途说明
- 交易类型标记为"到账"

**FR-5.2 支出报销申请**
- 科研人员提交支出报销申请
- 需关联具体预算科目
- 信息：项目编号、预算科目 ID、金额、日期、用途
- 提交后状态为"待审批"

**FR-5.3 支出报销审批**
- 财务处管理员审批支出申请
- 审批结果：通过 / 驳回
- 审批通过时，数据库触发器：
  - 检查预算余额是否充足
  - 余额不足则拦截并报错（`SIGNAL SQLSTATE '45000'`）
  - 余额充足则自动更新预算表的已支出金额

**FR-5.4 预算执行率监控**
- 展示项目各科目预算执行情况
- 汇总信息：总预算、总支出、综合执行率

**FR-5.5 结余经费处理**
- 支持两种处理方式：转入下一年 / 上缴学校

---

### 3.6 成果管理

**FR-6.1 成果登记**
- 科研人员登记项目产出的科研成果
- 成果类型：论文、专利、软件著作权、获奖、标准、成果转化
- 信息：项目编号、成果类型、标题、发表日期、作者、附件链接

**FR-6.2 成果查询**
- 按项目查询该项目下所有已登记的成果

---

### 3.7 统计报表

**FR-7.1 学院科研统计**
- 按年度统计各学院的项目申报数、立项数、立项率、到账经费总额
- 通过存储过程 `sp_college_project_stats` 实现

**FR-7.2 研究人员工作量考核**
- 按年度统计每位科研人员负责的项目总数、在研项目数、成果数量
- 通过存储过程 `sp_researcher_workload` 实现

**FR-7.3 成果类型分布统计**
- 按成果类型分组统计总数量

### 3.8 综合查询

**FR-8.1 项目列表**
- 按申请日期倒序展示所有项目
- 显示项目编号、名称、负责人、状态、验收日期

**FR-8.2 项目详情**
- 按项目编号查询单个项目的完整信息

**FR-8.3 即将到期项目**
- 查询 30 天内到期且未终止/验收通过的项目
- 计算并显示剩余天数

### 3.9 文件管理

**FR-9.1 文件上传**
- 支持通用文件上传
- 使用安全文件名处理（Werkzeug `secure_filename`）
- 文件名添加时间戳前缀避免冲突
- 上传后返回文件访问 URL

**FR-9.2 文件下载**
- 通过 URL 访问已上传的文件

## 4. 非功能性需求

### 4.1 性能需求

| 指标 | 要求 |
|------|------|
| 页面首次加载时间 | ≤ 3 秒 |
| API 响应时间 | ≤ 1 秒（简单查询）|
| 并发用户数 | 支持 50 人同时在线操作 |
| 数据库查询优化 | 关键查询字段建立索引 |

### 4.2 安全性需求

- **NFR-SEC-01**: 数据库密码等敏感配置不应硬编码在源码中，建议使用环境变量
- **NFR-SEC-02**: 所有 SQL 查询必须使用参数化查询，防止 SQL 注入
- **NFR-SEC-03**: 文件上传应校验文件类型和大小，防止恶意文件上传
- **NFR-SEC-04**: 用户密码应加密存储（当前为明文，待改进）
- **NFR-SEC-05**: 生产环境应关闭 Flask Debug 模式
- **NFR-SEC-06**: CORS 应配置允许的源白名单
- **NFR-SEC-07**: 预算超支由数据库触发器在底层拦截，确保数据完整性

### 4.3 可用性需求

- **NFR-USA-01**: 界面采用响应式设计（Bootstrap 5），适配不同屏幕尺寸
- **NFR-USA-02**: 操作结果应有明确的成功/失败提示
- **NFR-USA-03**: 表单应提供必填字段标识和输入提示
- **NFR-USA-04**: 预算预警以红色高亮显示，帮助用户快速识别风险

### 4.4 可维护性需求

- **NFR-MNT-01**: 代码应有清晰的中文注释标记功能模块
- **NFR-MNT-02**: 数据库脚本应包含完整的建库、建表、测试数据、存储过程等
- **NFR-MNT-03**: API 路由应遵循 RESTful 风格，命名语义化
- **NFR-MNT-04**: 统一 JSON 返回格式 `{"code": 200, "data": ...}`

### 4.5 数据完整性需求

- **NFR-DAT-01**: 所有外键关系在数据库层面定义
- **NFR-DAT-02**: 项目删除时级联删除关联的预算记录（`ON DELETE CASCADE`）
- **NFR-DAT-03**: 研究人员删除时其关联的学院外键设为 NULL（`ON DELETE SET NULL`）
- **NFR-DAT-04**: 经费支出自动更新预算已用金额，由触发器保证一致性

## 6. 接口需求

### 6.1 后端 API 接口

所有 API 路由前缀为 `/api`，返回格式为：

```json
{
  "code": 200,       // 200 成功, 400 请求错误, 500 服务器错误
  "data": [...],     // 列表数据
  "message": "..."   // 操作提示消息
}
```

## 7. 约束与假设

### 7.1 技术约束

- **TC-01**: 系统采用前后端一体化架构，前端代码与后端模板耦合在 `index.html` 中
- **TC-02**: 目前无用户认证与授权机制（session/token），所有操作未做权限校验
- **TC-03**: 数据库连接配置硬编码在 `app.py` 中
- **TC-04**: 前端 API 地址硬编码为 `http://127.0.0.1:5000/api`
- **TC-05**: 文件上传未限制文件类型和大小

### 7.2 业务假设

- **BA-01**: 每个项目只由一名科研人员担任负责人
- **BA-02**: 形式审查不通过的项目退回后可修改重新提交
- **BA-03**: 验收评审不通过的项目退回执行阶段，可再次申请验收
- **BA-04**: 预算支出申请审批通过后不可撤销
- **BA-05**: 项目一旦终止不可恢复

### 7.3 依赖约束

- 系统依赖 MySQL 数据库，需要预先安装和配置
- 前端依赖 CDN 加载 Bootstrap 和 jQuery，需要网络连接
- Python 依赖项通过 pip/uv 管理，见 `uploads/requirements.txt`
